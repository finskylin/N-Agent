# 需求文档规范

## 文档结构

```markdown
# [项目名称] 需求文档

## 1. 项目概述
- **项目背景**：为什么要做这个项目，解决什么问题
- **核心目标**：用一句话描述最终交付物
- **目标用户**：谁来用

## 2. 功能需求

### 2.1 功能清单
| 编号 | 功能名称 | 优先级 | 描述 |
|------|---------|--------|------|
| F-01 | 用户登录 | P0 | 支持账号密码登录 |
| F-02 | ... | P1 | ... |

优先级说明：
- P0：必须有，没有无法上线
- P1：应该有，影响核心体验
- P2：可以有，后续迭代

### 2.2 功能详细描述（每个功能必须覆盖以下 7 项）

#### F-01: [功能名称]
- **用户故事**：作为[角色]，我希望[功能]，以便[价值]
- **输入**：用户提供什么（数据格式、来源、必填/选填）
- **处理逻辑**：系统做什么（核心算法/流程，分步骤描述）
- **输出**：返回什么结果（数据结构、展示方式）
- **异常场景**：出错时怎么处理（每种错误的提示和恢复策略）
- **边界条件**：极端情况（空数据、超大数据、并发、超时）
- **验收标准**：怎样算做完了（可测试的具体条件，不是"正常工作"）

### 2.3 功能交互流程
[对于涉及多步骤交互的功能，用编号流程描述用户操作路径]

## 3. 非功能需求
- **性能**：响应时间（P99 < Xms）、并发量、数据量级
- **安全**：认证方式、数据加密、权限控制、敏感数据处理
- **可用性**：部署方式、容灾、监控告警、SLA 目标
- **兼容性**：浏览器/系统/设备要求
- **可扩展性**：未来可能的功能扩展方向

## 4. 技术约束
- **技术栈**：指定或推荐的语言/框架
- **运行环境**：Docker/本地/云服务器
- **数据存储**：数据库类型、文件存储
- **第三方依赖**：需要对接的外部服务/API
- **现有系统集成**：需要对接的内部系统
- **配置化要求**：所有可变值必须通过配置管理，禁止硬编码

## 5. 交互方式
- **入口**：API / CLI / Web UI / 钉钉 / 飞书
- **原型/线框图**：如有，附链接或描述

## 6. 里程碑
| 阶段 | 交付物 | 预期完成 |
|------|--------|---------|
| 需求评审 | 需求文档确认 | Day 0 |
| 技术设计 | 设计文档确认 | Day 1 |
| 开发 | 核心功能 | Day 2-3 |
| 测试 | 测试通过 | Day 4 |
| 上线 | 部署完成 | Day 5 |
```

## 文档保存规则

需求文档必须保存到项目中，路径规则：
```
{project_root}/docs/requirements/
├── PRD_{项目名称}_{日期}.md        # 需求文档
├── PRD_{项目名称}_{日期}_v2.md     # 需求变更版本
└── meeting_notes/                  # 需求评审记录
    └── {日期}_需求评审.md
```

## 需求评审环节（必须执行）

需求文档完成后，必须进行需求评审，输出评审结论：

```markdown
## 需求评审结论

### 评审检查项
| 检查项 | 状态 | 说明 |
|--------|------|------|
| 功能目标是否清晰 | ✅/❌ | |
| 每个功能是否有详细的输入/输出/异常/验收标准 | ✅/❌ | |
| 技术约束是否完整 | ✅/❌ | |
| 非功能需求是否明确 | ✅/❌ | |
| 是否有遗漏的边界条件 | ✅/❌ | |
| 是否有冲突的需求项 | ✅/❌ | |
| 配置化要求是否覆盖所有可变值 | ✅/❌ | |

### 风险识别
| 风险 | 影响 | 建议 |
|------|------|------|
| ... | ... | ... |

### 评审结论
- [ ] 通过，可进入技术设计阶段
- [ ] 需补充以下信息后再评审：[列出待补充项]
- [ ] 需求存在重大问题，需重新讨论：[列出问题]
```

**评审流程**：
1. Agent 输出需求文档
2. Agent 自检评审检查项
3. 输出评审结论，询问用户："需求评审完成，以上是否有需要补充或修改的？确认后进入技术设计阶段。"
4. 用户确认后，需求文档定稿

## 需求确认检查清单

收到需求后，逐项确认以下内容，有疑问的必须向用户澄清：

```
☐ 核心功能目标是否清晰？能用一句话说清楚吗？
☐ 每个功能是否有详细的输入/输出/异常/验收标准？
☐ 技术栈是否确定？（语言/框架/数据库）
☐ 运行环境是否明确？（Docker/本地/服务器）
☐ 交互方式是否明确？（API/Web/CLI/钉钉）
☐ 是否有性能要求？（并发/响应时间/数据量）
☐ 是否需要集成现有系统？
☐ 所有可变值是否明确需要配置化？
```

## 需求不明确时的处理

当用户需求模糊时，输出以下模板让用户补充：

```
## 需求确认

**功能目标**：[你理解的核心功能]

**待确认项**：
1. [具体问题1]
2. [具体问题2]
3. ...

**我的推荐方案**：[简述推荐技术方案和理由]

请确认以上信息，或直接说"按推荐方案来"。
```
