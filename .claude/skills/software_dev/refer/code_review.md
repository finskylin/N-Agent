# 代码评审规范

## 评审时机

每完成一个模块的开发后，必须进行自检。不是全部写完再检查，而是**边写边检查**。

## 铁律检查（最高优先级）

以下违规必须立即修复，不允许任何例外：

```
☐ 【禁止硬编码】是否有硬编码的 URL/端口/密钥/API Key/模型名称/阈值？
  → 全部改为从 .env / config 文件读取
☐ 【禁止 Mock】是否有 mock 数据或假实现（非测试代码中）？
  → 必须接真实 API/数据库/配置文件
☐ 【配置化】所有可变值是否通过配置管理？
  → URL、端口、阈值、开关、业务规则全部配置化
☐ 【.env.example】是否同步更新了配置模板？
  → 每增加一个环境变量必须同步写入 .env.example
```

## 代码质量检查清单

### 安全性
```
☐ 是否有硬编码的 URL/端口/密钥/API Key？（铁律：必须配置化）
☐ 是否有 SQL 注入风险？（使用参数化查询）
☐ 是否有 XSS 风险？（用户输入是否转义）
☐ 是否有明文存储密码？（必须 hash）
☐ 是否有敏感信息写入日志？
☐ 是否有敏感信息提交到代码仓库？（.env 必须 gitignore）
```

### 健壮性与鲁棒性
```
☐ 所有外部调用（HTTP/DB/文件IO）是否有 try-except？
☐ HTTP 请求是否设置了超时？（默认超时 ≤ 30s）
☐ 是否有重试机制？（外部 API 调用至少重试 2 次）
☐ 是否有降级/兜底方案？（外部服务不可用时返回合理默认值）
☐ 异常是否记录了日志？（ERROR 级别 + 上下文信息）
☐ 是否有未关闭的文件/连接/资源？（使用 with 语句）
☐ 是否处理了空值/None 的情况？
☐ 是否处理了并发/竞争条件？（数据库操作、文件操作）
```

### 扩展性
```
☐ 核心逻辑是否依赖具体实现？（应依赖接口/抽象）
☐ 新增功能是否需要修改已有代码？（开闭原则）
☐ 业务规则是否硬编码在代码里？（应配置驱动）
☐ 数据源/通知渠道等是否可通过配置切换？
```

### 可维护性
```
☐ 函数/方法是否超过 50 行？（超过则拆分）
☐ 是否有重复逻辑可以提取为公共函数？
☐ 变量/函数命名是否清晰？（能看懂在做什么）
☐ 是否有 TODO/FIXME 未处理？
☐ 配置是否外置？（.env / config 文件）
```

### 规范性
```
☐ REST API 是否有统一响应格式？ {"code": 0, "data": ..., "message": "ok"}
☐ 日志是否使用标准库？（loguru / logging）
☐ 关键操作是否有 INFO 日志？
☐ 错误是否有 ERROR 日志（含上下文：请求参数、用户ID、时间戳）？
☐ 数据库查询频繁的字段是否有索引？
☐ API 接口是否有参数校验？（Pydantic / Joi / Zod）
```

## 后端代码规范

### Python
- **配置管理**：环境变量 → Pydantic Settings / os.getenv，禁止硬编码
- **错误处理**：所有外部调用必须 try-except + 重试 + 降级
- **日志**：使用 loguru 或标准 logging，ERROR 日志必须含上下文
- **API 响应**：统一格式 `{"code": 0, "data": ..., "message": "ok"}`
- **异步**：IO 密集型操作使用 async/await
- **类型注解**：函数参数和返回值加类型注解
- **超时**：所有 HTTP 请求设置 timeout（默认 ≤ 30s）

### Node.js
- **配置管理**：dotenv + process.env，禁止硬编码
- **错误处理**：try-catch + express error middleware + 重试
- **日志**：winston / pino
- **API 响应**：统一格式

## 前端代码规范

- **框架**：优先 Vue 3 + Vite 或 React，简单页面可用原生 HTML+JS
- **状态管理**：复杂页面用 Pinia (Vue) / Zustand (React)，简单页面直接 ref/useState
- **API 调用**：统一封装 axios/fetch，含请求拦截（加 token）和响应拦截（统一错误处理）
- **样式**：优先 Tailwind CSS 或 Element Plus，禁止内联 style 堆砌
- **组件化**：可复用的 UI 抽成独立组件
- **环境配置**：API 地址等通过 .env 或 config 文件管理，禁止硬编码

## 数据库规范

- 优先 SQLite（本地/单机应用），不要为简单需求引入 MySQL/Redis
- Schema 变更必须有迁移脚本（alembic / 手写 SQL）
- 查询频繁的字段必须建索引
- 禁止 SELECT *，明确指定字段
- 大表操作加 LIMIT
- 连接池配置化（最大连接数、超时时间）

## 发现问题的处理

- 发现问题**当场修复**，不遗留到下一步
- 修复后重新运行相关测试确认
- 如果修复涉及设计变更，需要通知用户
- 铁律违规（硬编码/Mock/未配置化）发现一处修复一处，不允许"先记录后修"
