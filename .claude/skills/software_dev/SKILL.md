## software_dev — 项目开发全流程规范

本 skill 在 LLM 首次调用时注入，指导 Agent 按规范流程推进软件开发任务。

详细规范文档位于 `refer/` 目录，按阶段拆分（执行到对应阶段时用 `bash cat` 读取）：

| 阶段 | 规范文件 | 核心要求 |
|------|---------|---------|
| 需求确认 | `/app/.claude/skills/software_dev/refer/requirements.md` | 输出需求文档 → 需求评审 → 用户确认 |
| 技术设计 | `/app/.claude/skills/software_dev/refer/technical_design.md` | 按标准目录结构输出设计文档 → 用户确认 |
| 代码评审 | `/app/.claude/skills/software_dev/refer/code_review.md` | 铁律检查（禁止硬编码/Mock）+ 质量自检 |
| 上线部署 | `/app/.claude/skills/software_dev/refer/deployment.md` | 部署前检查 → 测试验证 → 验收报告 |

## 执行流程（必须严格遵守顺序）

```
1. 需求确认 → 输出需求文档（功能详细描述、验收标准）
   ↓
2. 需求评审 → 自检评审检查项 → 输出评审结论 → 等用户确认
   ↓ 用户确认
3. 技术设计 → 按标准目录结构输出设计文档（配置化设计、扩展性设计、鲁棒性设计）
   ↓ 用户确认
4. 开发实现 → 在 docker_operator (toolbox) 中逐步编码
   ↓ 每个模块完成
5. 代码自检 → 铁律检查 + 质量检查，发现问题当场修复
   ↓ 全部完成
6. 测试验证 → 启动测试 + E2E 测试 + 边界测试
   ↓ 测试通过
7. 部署上线 → docker-compose + 启动命令 + 验收报告
```

**例外**：用户说"直接做"、"按你的判断"、"快速实现"时，可跳过需求确认，但必须输出方案概述。

---

## 铁律（贯穿全流程）

1. **禁止硬编码**：所有 URL、端口、API Key、阈值、模型名称通过 .env / config 管理
2. **禁止 Mock**：非测试代码禁止 mock 数据，必须接真实 API/数据库
3. **配置化优先**：功能行为、业务规则、阈值、开关通过配置文件控制
4. **扩展性设计**：核心模块预留扩展点，新增功能不修改已有代码
5. **鲁棒性设计**：所有外部依赖必须有超时、重试、降级、兜底方案
6. **使用 docker_operator**：所有编码、脚本执行、软件安装必须在 toolbox 中完成
7. **必须测试**：每个功能必须有真实调用链的 E2E 测试

## 文档保存规则（所有文档必须持久化到项目中）

```
{project_root}/docs/
├── requirements/                # 需求文档
│   ├── PRD_{项目名称}_{日期}.md
│   └── meeting_notes/           # 需求评审记录
├── design/                      # 技术设计文档
│   ├── TDD_{项目名称}_{日期}.md
│   └── diagrams/                # 架构图/流程图
└── deployment/                  # 部署文档
    └── deploy_guide.md
```
